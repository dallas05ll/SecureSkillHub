## Patterns Found

### 1. The Root Cause of manual_review is Overwhelmingly "Agent B Missed Findings"

9 of 10 skills were flagged by Agent E with `compromised_agent_suspicion: "Agent B missed scanner findings; possible analysis gap."` This is NOT a compromised agent — it is a systematic issue where Agent B (LLM code analysis) only analyzes a subset of files while Agent C* (deterministic scanner) scans everything. The consistency check flags this discrepancy as suspicious.

**Impact:** This pattern inflates the manual_review queue. Skills with legitimate large codebases (documentation, tests, examples) will always show this discrepancy because Agent B focuses on source code while C* scans all files.

**Recommendation:** Consider updating the consistency check threshold. Agent B analyzing 50% or more of total files should not automatically flag as suspicious. The current system is over-triggering on docs-heavy repos.

### 2. The Score=100 → manual_review Paradox (find-skills)

find-skills scored 100/100 but was sent to manual_review because Agent D's rule requires "score >= 80 AND no critical/high findings" for auto-pass, and 1 high finding existed (localhost:3000 in a test file). This is a borderline false positive case.

**Recommendation:** Test file paths should receive lower severity weighting. A `localhost:` URL in a file path containing `test/` or `.test.` should not block auto-pass for a score=100 skill.

### 3. Score=70 Skills Fall Into Two Distinct Categories

Four skills scored 70: agent-browser, superpowers, seo-audit, OpenTofu. Analysis shows they are NOT equivalent:
- **Legitimate complex tools** (superpowers, OpenTofu): High finding counts are false positives from docs/tests. These should PASS.
- **Genuinely borderline tools** (agent-browser, seo-audit): High finding counts reflect real complexity and lower-trust signals. These should KEEP manual_review.

Score alone is insufficient to distinguish these. Risk context (owner trust, finding source location, obfuscation type) matters enormously.

### 4. "Obfuscation Detected!" Does Not Mean Actual Obfuscation

7 of 10 skills have obfuscation detections. In every case reviewed in detail, the "obfuscation" is:
- Base64 in crypto/encryption code (OpenTofu, playwright-mcp)
- Long separator strings matching long_base64_literal regex (superpowers)
- btoa() for standard browser auth (playwright-mcp)
- Base64 in test cert fixtures (playwright-mcp)

The `obfuscation_high_risk_count` field (which triggers instant-fail) is 0 for all 10 skills. The regular `obfuscation_count` is causing false alarm noise.

**Recommendation:** Add a `possible_false_positive` flag to the obfuscation category when: (a) the pattern appears in test files, (b) it appears in encryption-related files, or (c) the matched pattern is a standard crypto operation (base64.b64encode, btoa).

### 5. Score=85 Skills with Consistent Agent B Failures Cluster Around High-Volume Codebases

frontend-design, upstash--context7-mcp, browser-use, playwright-mcp all scored 85. The differentiation is:
- browser-use: 810 findings, 43 obfuscation → KEEP
- frontend-design: 154 findings, 4 obfuscation, 27 dangerous_calls in recalc.py → KEEP
- upstash/playwright-mcp: 47-66 findings, 2 obfuscation, reputable owners → PASS

Volume and owner trust are the deciding factors at the 85-score tier.
